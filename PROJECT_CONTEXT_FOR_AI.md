# AstraNav ROS 2 Project Context

This document is a self-contained handoff for explaining this project to an AI without giving it the whole repository. It summarizes the workspace structure, each package, launch flow, robot model, navigation stack, custom Python nodes, important topics, commands, and the most important code snippets.

## High-Level Summary

This is a ROS 2 workspace for a simulated differential-drive mobile robot called `astranav`. The project uses Gazebo/Ignition simulation, `ros2_control`, joystick teleoperation, SLAM Toolbox, Nav2 localization/navigation, QR detection with OpenCV, and custom docking/patrolling scripts.

The real source code lives under `src/`. The top-level `build/`, `install/`, and `log/` directories are generated by `colcon` and are not hand-written source.

Main capabilities:

- Spawn AstraNav in Gazebo worlds such as `small_house` and `qr_maze`.
- Publish robot state from a Xacro/URDF model.
- Drive a two-wheel differential robot using `diff_drive_controller`.
- Bridge Gazebo `/scan`, `/imu`, `/camera/image_raw`, `/camera/camera_info`, and `/clock` into ROS 2.
- Use joystick input through `joy`, `joy_teleop`, and `twist_mux`.
- Build maps with `slam_toolbox`.
- Localize with Nav2 AMCL against saved maps.
- Navigate using Nav2 planner, controller, behavior tree, smoother, and waypoint follower.
- Detect QR codes from the camera.
- Solve a QR-guided maze.
- Perform undocking, waypoint patrol, return-to-dock, QR alignment, and final docking.
- Trigger return-to-dock early when a simulated battery topic reports low battery.

#### `maze_solver.py`

Purpose: QR-directed maze behavior.

Inputs:

- `/camera/image_raw`
- `/scan`
- `/imu/out`

Output:

- `/wheel_controller/cmd_vel_unstamped`

States:

- `FORWARD`: drive forward at `0.15 m/s`.
- `TURNING`: rotate left or right based on QR command.
- `STOPPED`: stop permanently.

QR commands:

- `left`
- `right`
- `stop`

Main state machine:

```python
if self.state == "FORWARD":
    vel.linear.x = 0.15
    if self.front_distance <= 0.45:
        if self.qr_command == "stop":
            self.state = "STOPPED"
        elif self.qr_command in ["left", "right"]:
            self.state = "TURNING"
            self.target_yaw = self.calculate_target_yaw()
        else:
            vel.linear.x = 0.0

elif self.state == "TURNING":
    vel.linear.x = 0.0
    vel.angular.z = 0.3 if self.qr_command == "left" else -0.3
    yaw_diff = self.normalize_angle(self.target_yaw - self.current_yaw)
    if abs(yaw_diff) < 0.1:
        self.state = "FORWARD"
        self.qr_command = None
```

#### `auto_dock_undock.py`

Purpose: simple scripted undock and dock sequence.

Inputs:

- `/camera/image_raw`
- `/imu/out`
- `scan`

Output:

- `/cmd_vel`

Because it publishes `/cmd_vel`, it is intended to feed the controller through `twist_mux`, not directly to the wheel controller.

Algorithm:

1. Move backward for a fixed distance.
2. Rotate 180 degrees.
3. Rotate back to 0 radians.
4. Wait until a QR marker is detected.
5. Align the camera center with the QR marker center.
6. Drive forward slowly until front lidar distance is below `0.15 m`.

Key alignment logic:

```python
error = self.qr_center_x - (self.image_width / 2)
if abs(error) < 5:
    break
angular_speed = -error * 0.003
self.velocity_publisher(0.0, angular_speed)
```

Final docking:

```python
if self.front_distance < self.docking_threshold:
    break

if self.qr_detected and self.qr_center_x is not None:
    error = self.qr_center_x - (self.image_width / 2)
    angular_correction = -error * 0.002
else:
    angular_correction = 0.0

self.velocity_publisher(0.05, angular_correction)
```

#### `docling_with_patrolling.py`

Despite the name typo, this is a docking + patrolling script. It combines manual velocity commands, QR-based docking, and Nav2 waypoint navigation.

Inputs:

- `/camera/image_raw`
- `/imu/out`
- `scan`

Output:

- `/cmd_vel`

Uses:

- `BasicNavigator` from `nav2_simple_commander`.

Main sequence:

1. Set initial pose at dock:
   - `x = 1.5`
   - `y = 5.18`
   - `yaw = 1.57`
2. Undock by moving backward `0.6 m`.
3. Rotate.
4. Navigate to waypoints:
   - `(5.0, 1.0, 0.0)`
   - `(0.0, 3.0, 0.0)`
5. Return to a pre-dock pose:
   - `x = initial_x`
   - `y = initial_y - 0.4`
   - `yaw = initial_yaw`
6. Wait for QR detection.
7. Align with QR.
8. Drive forward until lidar says distance is under `0.15 m`.

Pose creation helper:

```python
def create_pose_stamped(self, x, y, yaw):
    q_x, q_y, q_z, q_w = quaternion_from_euler(0.0, 0.0, yaw)
    pose = PoseStamped()
    pose.header.frame_id = 'map'
    pose.header.stamp = self.nav.get_clock().now().to_msg()
    pose.pose.position.x = x
    pose.pose.position.y = y
    pose.pose.orientation.z = q_z
    pose.pose.orientation.w = q_w
    return pose
```

#### `auto_docking_with_battery.py`

Purpose: battery-aware version of docking + patrol.

Inputs:

- `/camera/image_raw`
- `/imu/out`
- `/scan`
- `/battery_status` as `std_msgs/msg/Int32`

Output:

- `/cmd_vel`

Main difference from `docling_with_patrolling.py`:

- Uses `MultiThreadedExecutor`.
- Stores battery with a thread lock.
- Cancels waypoint navigation when battery is critically low.

Battery callback:

```python
self.battery_sub = self.create_subscription(Int32, '/battery_status', self.battery_callback, 10)

def battery_callback(self, msg):
    with self.battery_lock:
        self.battery = msg.data
```

Low-battery cancellation:

```python
while not self.nav.isTaskComplete():
    current_battery = self.get_battery()

    if current_battery <= 5:
        self.get_logger().warn(
            f"Battery critically low ({current_battery}%)! Canceling navigation..."
        )
        self.nav.cancelTask()
        time.sleep(1.0)
        return False
```

## Important Runtime Assumptions

- This project appears designed for ROS 2 with Ignition/Gazebo integration packages:
  - `ros_gz_sim`
  - `ros_gz_bridge`
  - `ros_gz_image`
  - `ign_ros2_control`
- Nav2 packages are required:
  - `nav2_amcl`
  - `nav2_map_server`
  - `nav2_lifecycle_manager`
  - `nav2_controller`
  - `nav2_planner`
  - `nav2_smoother`
  - `nav2_behaviors`
  - `nav2_bt_navigator`
  - `nav2_waypoint_follower`
  - `nav2_simple_commander`
- OpenCV and cv_bridge are required for camera and QR scripts.
- `numpy` is used in the scripts but is not explicitly listed in `astranav_scripts/package.xml`.
- `tf_transformations` is used by navigation/docking scripts and is declared in `astranav_navigation`, but not in `astranav_scripts/package.xml`.
- GUI scripts call `cv2.imshow`, so they need an environment with display support.

## Suggested Explanation For Another AI

Explain the project as a layered ROS 2 mobile robot simulation:

1. `astranav_description` defines the robot and simulation world. The robot is a compact differential-drive robot with two driven wheels, caster supports, lidar, camera, and IMU. Gazebo plugins simulate the sensors and `ros2_control`.
2. `astranav_controller` connects the simulated wheel joints to `diff_drive_controller`. It also provides joystick and `twist_mux` support so joystick, keyboard, navigation, and docking commands can be prioritized.
3. `astranav_mapping` runs SLAM Toolbox to build maps from `/scan` and odometry.
4. `astranav_localization` loads a saved map and uses AMCL to estimate robot pose.
5. `astranav_navigation` runs Nav2. It uses a Smac 2D planner, regulated pure pursuit controller, obstacle/static/inflation costmaps, a simple behavior tree with replanning and recovery, and waypoint follower support.
6. `astranav_bringup` combines all these into complete demo launch files.
7. `astranav_scripts` contains custom behaviors:
   - sensor debug readers,
   - QR code detection,
   - QR-commanded maze solving,
   - simple dock/undock,
   - waypoint patrol plus docking,
   - battery-aware patrol cancellation and return-to-dock.

## Potential Issues Or Things To Improve

- Most package descriptions and licenses are still `TODO`.
- `astranav_scripts/package.xml` does not list every Python dependency used by scripts, such as `numpy`, `tf_transformations`, `std_msgs`, and `nav2_simple_commander`.
- `astranav_navigation` installs `waypoint_following.py` as a program but does not define a console script entry point because it is an `ament_cmake` package using `ament_python_install_package`.
- `maze_solver.py` publishes directly to `/wheel_controller/cmd_vel_unstamped`, bypassing `twist_mux`. That is okay for a maze-specific mode, but it means joystick/Nav2 priority handling is bypassed.
- `auto_dock_undock.py`, `docling_with_patrolling.py`, and `auto_docking_with_battery.py` all define a class named `SimpleDockingNode`; names are local to each module, but a clearer class name per script would improve readability.
- Some docking scripts use time-based dead-reckoning movement (`distance / speed`) rather than odometry-based movement, so real-world accuracy may drift.
- `cv2.imshow` is used in callbacks or loops, so headless machines may fail unless display forwarding is available.
- The bringup file comments manually switch between SLAM and localization/navigation. A launch argument like `mode:=slam|nav` would make this cleaner.
